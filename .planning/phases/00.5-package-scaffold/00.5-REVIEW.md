# Phase 0.5: Code Review Report

**Reviewed:** 2026-04-28T00:00:00Z
**Depth:** standard
**Files Reviewed:** 7
**Status:** issues_found

## Summary

Phase 0.5 migrated all six SKILL.md files from stale `python skills/*/scripts/*.py` invocations to `uv run <entry-point>` form and introduced a `[dependency-groups]` dev section in `pyproject.toml`. The migration to `uv run` is structurally correct across all skills — every named entry point (`init-project`, `refine`, `req-ops`, `review`, `report`, `meeting`) is declared in `[project.scripts]` and resolves to a `main()` function in the source package.

Two critical issues were found: `pytest` and `ruff` are declared as production runtime dependencies rather than dev/test tooling, and `prek` is declared as a production dependency despite not being imported anywhere in the package source. Five warnings cover SKILL.md command documentation that diverges from actual CLI argument behaviour, which will cause agent hallucination when the agent follows the documented syntax.

## Critical Issues

### CR-01: `pytest` and `ruff` declared as runtime production dependencies

**File:** `pyproject.toml:13-14`
**Issue:** Both `pytest>=9.0.3` and `ruff>=0.15.11` are listed in `[project.dependencies]` — the mandatory runtime install set. A `[dependency-groups]` dev group was added in this phase (lines 38-45) but `pytest` and `ruff` were not moved into it. Any downstream consumer who installs `requirements-agent` as a library (e.g., via `pip install requirements-agent`) pulls in a test framework and a linter as hard runtime requirements. This inflates the install footprint by ~50 MB and violates PEP 508 / packaging best practice, which classifies these as development-only tools.
**Fix:**
```toml
# Remove from [project.dependencies]:
#   "pytest>=9.0.3",
#   "ruff>=0.15.11",

# Add to [dependency-groups]:
[dependency-groups]
dev = [
    "lazydocs>=0.4.8",
    "pre-commit>=4.6.0",
    "ty>=0.0.32",
    "bandit>=1.9.4",
    "interrogate>=1.7.0",
    "pytest>=9.0.3",
    "ruff>=0.15.11",
]
```
Also update `CLAUDE.md` commands to use `uv run --group dev pytest` / `uv run --group dev ruff` if isolation is desired, though `uv run pytest` already resolves from the dev group when invoked inside the project.

### CR-02: `prek` declared as a production dependency but never imported

**File:** `pyproject.toml:11`
**Issue:** `prek>=0.3.9` appears in `[project.dependencies]`. `prek` is a Rust-based git-hook runner advertised as a drop-in alternative to `pre-commit`. It is not imported anywhere in `src/requirements_agent_tools/`, not referenced in `hooks/`, not referenced in any YAML config, and serves no runtime purpose. Meanwhile `pre-commit>=4.6.0` is correctly placed in `[dependency-groups.dev]`. Shipping `prek` as a runtime dependency installs an unnecessary native binary for every production install and creates two redundant hook-runner dependencies.
**Fix:**
```toml
# Remove from [project.dependencies]:
#   "prek>=0.3.9",

# If prek is intended to replace pre-commit for hook running, move it:
[dependency-groups]
dev = [
    ...
    "prek>=0.3.9",
    # "pre-commit>=4.6.0",  # remove whichever is not used
]
```
Confirm which hook runner is actually wired up (check `.pre-commit-config.yaml` or equivalent prek config) and keep only one.

## Warnings

### WR-01: `meeting integrate` command documents `--all-unintegrated` and `--status-summary` as required, but both are optional in the CLI

**File:** `skills/meeting-agent/SKILL.md:43-46`
**Issue:** The documented invocation is:
```bash
uv run meeting integrate [--project <slug>] \
  --all-unintegrated \
  --status-summary "<updated project narrative>"
```
Square-bracket notation `[...]` is used only for `--project`. This implies `--all-unintegrated` and `--status-summary` are mandatory. In `meeting.py`, both arguments have defaults (`action="store_true"` with an implicit `False`, and `default=None` respectively). Neither is `required=True`. Additionally, the `--meeting-ids` option (comma-separated meeting IDs for targeted integration) exists in the code (`meeting.py:225`) but is entirely absent from the documentation.
**Fix:**
```markdown
uv run meeting integrate [--project <slug>] \
  [--all-unintegrated] \
  [--meeting-ids "MTG-XXX,MTG-YYY"] \
  [--status-summary "<updated project narrative>"]
```

### WR-02: `meeting log` documents `--source` as required, but it defaults to `"other"` in the CLI

**File:** `skills/meeting-agent/SKILL.md:22`
**Issue:** The documented invocation shows:
```bash
uv run meeting log ... --source teams|slack|direct|email|zoom|in-person|other \
```
without square brackets, implying it is required. In `meeting.py:193`, `--source` has `default="other"`, making it optional. An agent following the documentation will pass `--source` unconditionally, which works, but an agent troubleshooting a missing-argument error will be misled.
**Fix:**

```markdown
[--source teams|slack|direct|email|zoom|in-person|other] \
```

### WR-04: `meeting update_decision` subcommand name uses underscore but agent-facing docs should note it

**File:** `skills/meeting-agent/SKILL.md:37`
**Issue:** The SKILL.md documents `uv run meeting update_decision` (underscore). The CLI parser registers the subcommand as `update_decision` (confirmed in `meeting.py:214`). While the CLI accepts this correctly, this deviates from standard CLI convention where subcommands use hyphens (`update-decision`). More critically, the SKILL.md is inconsistent: all other subcommands use hyphens (`close_action` also uses underscore, `list`, `get`, `decisions` use neither), but the general style in this codebase is hyphen-separated. If a future refactor normalises to hyphens in the parser, every agent prompt that has cached `update_decision` will break silently.
**Fix:** Align the parser and documentation. Either normalise all subcommands to hyphens in the parser and update SKILL.md, or add an explicit note in the SKILL.md warning that these two subcommands use underscores:
```python
# In meeting.py build_parser():
ud = sub.add_parser("update-decision")  # hyphen-normalised
ca = sub.add_parser("close-action")
```
And update SKILL.md to match.

## Info

### IN-01: `ty>=0.0.32` pinned to a pre-release version; `0.0.33` already released

**File:** `pyproject.toml:44`
**Issue:** `ty` (Astral's type checker) is pinned at `>=0.0.32`. Version `0.0.33` is the current PyPI release as of review date. The `>=` lower bound means uv will resolve to `0.0.33` on a fresh sync, so this is not a functional bug. However, `ty` is a pre-release tool (`0.0.x`) and its API may change between patch versions. The `[tool.ty.rules]` config section suppresses `invalid-argument-type` as a `warn`, which is reasonable for sqlite3.Row access. No action strictly required — document the pre-release risk in a comment.
**Fix:**
```toml
[dependency-groups]
dev = [
    ...
    "ty>=0.0.32",  # pre-release; API may change on each 0.0.x bump
    ...
]
```

### IN-02: `new-project-initiation` directory name does not match the skill's `name` field or `agent.yaml` entry

**File:** `skills/new-project-initiation/SKILL.md:2`
**Issue:** The skill directory is `skills/new-project-initiation/` but the `name` frontmatter field is `project-init` and `agent.yaml` registers it as `project-init`. This three-way inconsistency (directory name vs SKILL.md name vs agent.yaml) is tolerated only because the gitagent framework resolves skills by `name` field, not directory name. The mismatch is a latent confusion risk for contributors following the `CLAUDE.md` convention ("Create `skills/<name>/SKILL.md`").
**Fix:** Rename the directory to match the canonical name:
```bash
git mv skills/new-project-initiation skills/project-init
```

### IN-03: `[tool.interrogate] ignore-semiprivate = false` will flag all `_`-prefixed methods

**File:** `pyproject.toml:54`
**Issue:** `ignore-semiprivate = false` means `interrogate` will require docstrings on single-underscore (`_`) prefixed methods. The codebase has several `_cli_io.py`, `_serialization.py`, and `_logging.py` modules with `_`-prefixed helpers that may lack docstrings, potentially causing CI to fail the `fail-under = 90` threshold immediately after the tool is wired up. This is not a correctness bug but will surface as an unexpected CI failure.
**Fix:** Either set `ignore-semiprivate = true` to match the `ignore-private = true` convention already in place, or audit all `_`-prefixed methods first to confirm they have docstrings:
```toml
[tool.interrogate]
ignore-semiprivate = true  # consistent with ignore-private = true
```
