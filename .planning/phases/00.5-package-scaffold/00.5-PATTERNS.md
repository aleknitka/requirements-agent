## Pattern Assignments

### All 6 `skills/*/SKILL.md` files (config, request-response / CRUD)

**Canonical analog:** `skills/meeting-agent/SKILL.md`

The meeting-agent SKILL.md is the most complete in the repo: full YAML frontmatter with all required fields, numbered prose sections, and a bash code block per sub-command. All SKILL.md updates must match this structure.

**Frontmatter pattern** (SKILL.md lines 1-11):
```yaml
---
name: meeting-agent
description: >
  Log meeting minutes, extract and track decisions, manage action items, and integrate
  meeting summaries into project status. Use this skill when the user wants to ...
license: MIT
allowed-tools: Read Edit Grep Glob
metadata:
  author: aleksander nitka
  version: "1.0.0"
  category: requirements
---
```

Rules extracted from the analog:
- `description` uses the YAML block scalar `>` (folded) — keep on one logical line, wrap freely.
- `allowed-tools` is a space-separated list of Claude tool names (not a YAML list).
- `metadata.version` is a quoted string (`"1.0.0"` not `1.0.0`).
- `license: MIT` is required. `review-requirements/SKILL.md` currently omits `license` and `allowed-tools` — both must be added.

**Commands section pattern** (SKILL.md lines 18-45):

The commands block must:
1. Use a fenced `bash` code block.
2. Document every sub-command on its own line with a brief inline comment.
3. Show all required flags and all optional flags in `[square brackets]`.
4. Use exactly `uv run <entry-point> <subcommand>` — never `python skills/...`.

Example from meeting-agent (to copy as structural template):
```bash
# Log
uv run meeting log \
  --title "..." --by "<agent/user>" \
  --source teams|slack|direct|email|zoom|in-person|other \
  [--source-url "..."] [--occurred-at "YYYY-MM-DD"] \
  [--attendees '["Alice","Bob"]'] \
  [--summary "..."] [--raw-notes "..."] \
  [--decisions '[{"title":"...","made_by":["Alice"],"status":"open","affects_reqs":["REQ-DATA-XXXX"]}]'] \
  [--action-items '[{"description":"...","owner":"Alice","due_date":"2025-04-10"}]']

# Read
```

**Complete `.pre-commit-config.yaml`** to create verbatim:
```yaml
repos:
  # --- Format (runs before lint) ---
  - repo: https://github.com/astral-sh/ruff-pre-commit
    rev: v0.15.12
    hooks:
      - id: ruff-format
      - id: ruff-check
        args: [--fix]

  # --- Credentials ---
  - repo: https://github.com/pre-commit/pre-commit-hooks
    rev: v6.0.0
    hooks:
      - id: detect-private-key

  # --- Security ---
  - repo: https://github.com/PyCQA/bandit
    rev: 1.9.4
    hooks:
      - id: bandit
        args: [-r, src/]
        pass_filenames: false

  # --- Type checking (local — no official pre-commit repo for ty yet) ---
  - repo: local
    hooks:
      - id: ty
        name: ty type check
        entry: uv run --with ty ty check
        language: system
        types: [python]
        pass_filenames: false

  # --- Docstring coverage ---
  - repo: https://github.com/econchick/interrogate
    rev: 1.7.0
    hooks:
      - id: interrogate
        pass_filenames: false

  # --- Documentation regeneration (lazydocs → Markdown) ---
  - repo: local
    hooks:
      - id: lazydocs
        name: regenerate API docs (lazydocs)
        entry: uv run --with lazydocs lazydocs --output-path docs/ requirements_agent_tools
        language: system
        pass_filenames: false
        always_run: true

  # --- Tests ---
  - repo: local
    hooks:
      - id: pytest
        name: pytest
        entry: uv run pytest
        language: system
        pass_filenames: false
        always_run: true
```

**Critical implementation notes (from RESEARCH.md):**
- `ruff-check` uses hook id `ruff-check` (not the legacy `ruff` id).
- `pass_filenames: false` on ty, interrogate, lazydocs, pytest — they need whole-project context.
- `bandit` overrides the PyCQA default (`pass_filenames: true`) to scan `src/` consistently.
- lazydocs uses `uv run --with lazydocs` so it runs inside the project venv where all deps are available.
- RESEARCH.md Pitfall 5 applies to lazydocs too: regenerated `docs/` files are not auto-staged. Developers must `git add docs/` after the hook runs, or the committed snapshot lags behind.
- RESEARCH.md Pitfall 2: interrogate hook will immediately fail every commit because current coverage is 51.2%. The planner must sequence docstring-writing tasks before enabling this hook at `fail-under = 90`. Options: start at `fail-under = 0` or comment the hook out until docstrings reach 90%.
- RESEARCH.md Pitfall 3: ty hook will error on `db/_serialization.py` unless `[tool.ty.rules] invalid-argument-type = "warn"` is set first (it is included in the pyproject.toml block above).
- RESEARCH.md Pitfall 4: bandit flags 3 medium-severity issues in existing code. Review with `uv run --with bandit bandit -r src/` before enabling.
