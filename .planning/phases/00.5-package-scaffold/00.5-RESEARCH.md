### Pattern 1: lazydocs — Markdown documentation generation

**What:** Generates GitHub-flavored Markdown from Python docstrings. Recursively follows subpackages. Must run inside the project venv (project deps are imported at doc-gen time).

**Why lazydocs over pdoc:** pdoc is HTML-only; it does not support Markdown output (GitHub issue closed as "not planned"). lazydocs natively generates `.md` files. D-09 was revised 2026-04-28 to use lazydocs. [VERIFIED: CONTEXT.md D-09 revision note; PATTERNS.md Key decision section]

**CLI invocation (verified against actual codebase — Plan 02 output is 21 .md files):**

```bash
# Source: verified via Plan 02 execution — exact command that produced docs/
uv run --with lazydocs lazydocs \
  --output-path docs/ \
  requirements_agent_tools
```

Run from the repo root. The package name (not path) is the argument.

**Why `uv run --with lazydocs`:** lazydocs imports the package at doc-gen time. Running via `uvx lazydocs` (isolated env) fails with `ModuleNotFoundError` because project deps are not available. `uv run --with lazydocs` injects lazydocs into the project's own venv. [VERIFIED: Plan 02 implementation notes in STATE.md]

### Pattern 2: `.pre-commit-config.yaml` — complete pipeline

**Key principle for local hooks:** Use `language: system` + `entry: uv run --with <tool> <cmd>` + `pass_filenames: false`. This runs the tool inside the project's uv-managed venv. [VERIFIED: community pattern; pre-commit local hook docs]

**Why `pass_filenames: false` on several hooks:** Tools like ty, interrogate, lazydocs, and pytest operate on the full project, not individual staged files.

```yaml
# Source: verified against official repos for external hooks;
#         local hook pattern verified via uv pre-commit integration docs
repos:
  # --- Format (must run before lint) ---
  - repo: https://github.com/astral-sh/ruff-pre-commit
    rev: v0.15.12
    hooks:
      - id: ruff-format
      - id: ruff-check
        args: [--fix]

  # --- Credentials ---
  - repo: https://github.com/pre-commit/pre-commit-hooks
    rev: v6.0.0
    hooks:
      - id: detect-private-key

  # --- Security ---
  - repo: https://github.com/PyCQA/bandit
    rev: 1.9.4
    hooks:
      - id: bandit
        args: [-r, src/]
        pass_filenames: false

  # --- Type checking (local — no official pre-commit repo for ty yet) ---
  - repo: local
    hooks:
      - id: ty
        name: ty type check
        entry: uv run --with ty ty check
        language: system
        types: [python]
        pass_filenames: false

  # --- Docstring coverage ---
  - repo: https://github.com/econchick/interrogate
    rev: 1.7.0
    hooks:
      - id: interrogate
        pass_filenames: false

  # --- Documentation regeneration (lazydocs → Markdown) ---
  - repo: local
    hooks:
      - id: lazydocs
        name: regenerate API docs (lazydocs)
        entry: uv run --with lazydocs lazydocs --output-path docs/ requirements_agent_tools
        language: system
        pass_filenames: false
        always_run: true

  # --- Tests ---
  - repo: local
    hooks:
      - id: pytest
        name: pytest
        entry: uv run pytest
        language: system
        pass_filenames: false
        always_run: true
```

**Note on bandit args:** `bandit` from PyCQA's pre-commit hook passes filenames by default. Adding `args: [-r, src/]` and `pass_filenames: false` overrides this to scan the full `src/` tree consistently. [VERIFIED: PyCQA/bandit `.pre-commit-hooks.yaml`]

**Note on ruff hook IDs:** The hook IDs are `ruff-format` and `ruff-check` (not the legacy `ruff` id). [VERIFIED: astral-sh/ruff-pre-commit README]

### Anti-Patterns to Avoid

- **`uvx lazydocs ...` as pre-commit hook:** Fails because uvx runs in an isolated environment without project deps (loguru, pydantic, sqlite-vec not available). Use `uv run --with lazydocs lazydocs ...` instead.
- **`pdoc` for Markdown output:** pdoc is HTML-only. D-09 was revised to use lazydocs. Never reference pdoc in new plans or hooks.
- **`pass_filenames: true` on ty/interrogate/lazydocs/pytest:** These tools need whole-project context. Always `pass_filenames: false`.
- **Enabling interrogate hook before reaching 90% coverage:** Will immediately block every commit. Coverage must reach 90% (currently 50.6%) before the hook is activated.
- **Using `# nosec` without justification comment:** Bandit nosec suppressions must include the rule ID and reason, e.g. `# nosec B608 — column names are internal constants, not user input`.

## Common Pitfalls

### Pitfall 1: lazydocs runs in wrong environment
**What goes wrong:** `uvx lazydocs requirements_agent_tools` exits with `ModuleNotFoundError`.
**Why it happens:** `uvx` creates an isolated environment without project deps.
**How to avoid:** Always invoke as `uv run --with lazydocs lazydocs ...`.
**Warning signs:** Any `ModuleNotFoundError` or `ImportError` when lazydocs runs.

### Pitfall 2: interrogate coverage at 50.6% blocks commits immediately
**What goes wrong:** Enabling the interrogate hook with `fail-under = 90` immediately fails every commit.
**Why it happens:** 82 of 166 public API items lack docstrings (current state: 50.6%).
**How to avoid:** Write docstrings first (Wave A of Plan 03), then enable the hook (Wave B). The hook can be committed with `fail-under = 0` or commented out during the docstring-writing wave.
**Warning signs:** `RESULT: FAILED (minimum: 90.0%, actual: 50.6%)` on every commit.

### Pitfall 3: ty produces errors on existing code (sqlite3.Row access)
**What goes wrong:** `ty check src/` reports `invalid-argument-type` errors in `db/_serialization.py`.
**Why it happens:** sqlite-vec does not ship type stubs; ty infers `Any` from sqlite3.Row.
**How to avoid:** `[tool.ty.rules] invalid-argument-type = "warn"` is already set in pyproject.toml — hook will pass.
**Warning signs:** Commit blocked by type errors in `db/_serialization.py`.

### Pitfall 4: bandit flags 3 Medium issues in existing code
**What goes wrong:** The bandit hook fails on commit due to existing SQL construction patterns.
**Why it happens:** Three locations use f-string SQL construction flagged as B608 (hardcoded SQL expressions). All three are false positives: the dynamic portions are internal constants or parameterized values, not user input.
**How to avoid:** Add `# nosec B608` to the three flagged lines before enabling the hook. See Plan 03 research for exact locations.
**Warning signs:** Bandit output showing `B608: hardcoded_sql_expressions` on minutes.py:97 or requirements.py:201 or requirements.py:284.

### Pitfall 5: lazydocs pre-commit hook writes docs/ but changes aren't committed
**What goes wrong:** The lazydocs hook regenerates `docs/` on every commit, but the newly written files are not staged.
**Why it happens:** pre-commit hooks run after `git add` has captured the staging area. Files written by a hook during the commit are not automatically staged.
**How to avoid:** Run `uv run --with lazydocs lazydocs --output-path docs/ requirements_agent_tools && git add docs/` manually before committing, or document this in CLAUDE.md.
**Warning signs:** `git status` shows modified `docs/` files after a commit.

### Plan 03: Pre-commit Pipeline — Key Findings

#### Finding 1: The lazydocs hook invocation

The correct hook entry for `lazydocs` in `.pre-commit-config.yaml`:

```yaml
- repo: local
  hooks:
    - id: lazydocs
      name: regenerate API docs (lazydocs)
      entry: uv run --with lazydocs lazydocs --output-path docs/ requirements_agent_tools
      language: system
      pass_filenames: false
      always_run: true
```

**Why `always_run: true`:** Without it, lazydocs only runs when Python files are staged. If only config files or docs are changed, the hook skips — leaving docs stale. `always_run: true` guarantees docs are regenerated on every commit regardless of what is staged.

**Why no `--src-base-url`:** The PATTERNS.md draft included `--src-base-url https://github.com/org/requirements-agent/blob/main/` but the repo URL contains a placeholder `org`. Omit this flag for now — lazydocs works correctly without it and the output is still valid Markdown. [VERIFIED: Plan 02 output — 21 .md files generated without this flag]

**Does `always_run: true` create docs/ staging problems?** Yes — the same as Pitfall 5. The hook regenerates `docs/` but newly written files are NOT auto-staged by pre-commit. Two options for Plan 03 to handle this:
- Option A (recommended): Document in CLAUDE.md that developers must `git add docs/` after the lazydocs hook modifies docs, or run `uv run --with lazydocs lazydocs --output-path docs/ requirements_agent_tools && git add docs/` before committing.
- Option B: Move lazydocs to a pre-push hook (slower feedback, but avoids the staging problem entirely).
**Recommendation for planner: Option A** — same pattern used for other doc-generation tools in the ecosystem. The scaffold phase intent is to have a hook in place, not to solve the staging UX perfectly. [ASSUMED — pre-push alternative is viable but out of scope per D-15]

#### Finding 3: Bandit issues — exact locations and recommended fix

**Verified by `uv run --with bandit bandit -r src/` on 2026-04-29:**

All 3 issues are `B608: hardcoded_sql_expressions` (Medium severity):

1. `src/requirements_agent_tools/db/minutes.py:97` — f-string SQL with `{where}` clause. The `where` variable is constructed from hardcoded column filter strings (`occurred_at`, `minute_id`), not user input. Parameters are separately passed. **False positive.**

2. `src/requirements_agent_tools/db/requirements.py:201` — f-string SQL `UPDATE requirements SET {', '.join(sets)}`. The `sets` list contains hardcoded column names constructed from validated field names. **False positive.**

3. `src/requirements_agent_tools/db/requirements.py:284` — multi-line f-string SQL with `{tag_join}` and `{where}`. Both variables are constructed from hardcoded column/table names; parameters are separately passed. **False positive.**

**Recommended fix:** Add `# nosec B608` inline comment at each flagged line with a brief justification. This is the standard bandit pattern for confirmed false positives. Example:

```python
# minutes.py:97
rows = conn.execute(
    f"SELECT * FROM minutes{where} ORDER BY occurred_at",  # nosec B608 — where clause uses hardcoded column filters; values are parameterized
    params,
```

**Do NOT restructure the SQL:** These are well-structured parameterized queries. Refactoring them to avoid f-strings would reduce readability with no security benefit. The `# nosec B608` approach is correct. [VERIFIED: bandit docs — nosec is the intended escape hatch for false positives]

**No bandit config section needed in pyproject.toml** for this case — three inline `# nosec B608` comments are sufficient and more informative (they document the justification at the affected line). [ASSUMED — bandit pyproject.toml config section exists but is overkill for 3 targeted suppressions]
