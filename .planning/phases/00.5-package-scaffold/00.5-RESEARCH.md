### Pattern 1: pdoc — HTML documentation generation

**What:** Generates self-contained HTML from Python docstrings. Recursively follows subpackages. Must run inside the project venv (project deps are imported at doc-gen time).

**Critical constraint:** pdoc outputs HTML only. There is no `--format markdown` or similar flag. [VERIFIED: `uv run --with pdoc pdoc --help` — output format is not a CLI option; confirmed by pdoc issue #308 closed as "not planned"]

**CLI invocation (verified against actual codebase):**

```bash
# Source: verified via uv run --with pdoc pdoc --help + actual execution
uv run --with pdoc pdoc \
  --docformat google \
  src/requirements_agent_tools \
  -o docs/
```

This writes the full tree shown above. Run from the repo root.

**Why `uv run --with pdoc`:** pdoc imports the package at doc-gen time. Running via `uvx pdoc` (isolated env) fails with `ModuleNotFoundError: No module named 'loguru'` because project deps are not available. `uv run --with pdoc` injects pdoc into the project's own venv where loguru, pydantic, etc. are already installed. [VERIFIED: confirmed by testing both approaches]

**pyproject.toml — no `[tool.pdoc]` config section exists.** pdoc does not support pyproject.toml configuration. All options are CLI flags only. [VERIFIED: pdoc docs + `pdoc --help` — no `--config` flag]

**Google docstring style:** Pass `--docformat google` on the CLI (or set `__docformat__ = "google"` at module level as a per-module override). [VERIFIED: Context7 /mitmproxy/pdoc + `pdoc --help`]

### Pattern 2: `.pre-commit-config.yaml` — complete pipeline

**What:** Defines 8 hooks covering format, lint, security, types, coverage, docs, tests.

**Key principle for local hooks:** Use `language: system` + `entry: uv run --with <tool> <cmd>` + `pass_filenames: false`. This runs the tool inside the project's uv-managed venv. [VERIFIED: community pattern from astral-sh/ty#269 and uv pre-commit docs]

**Why `pass_filenames: false` on several hooks:** Tools like ty, interrogate, pdoc, and pytest operate on the full project, not individual staged files. Passing filenames would give them incomplete context and produce wrong results.

```yaml
# Source: verified against official repos for external hooks;
#         local hook pattern verified via uv pre-commit integration docs
repos:
  # --- Format (must run before lint) ---
  - repo: https://github.com/astral-sh/ruff-pre-commit
    rev: v0.15.12
    hooks:
      - id: ruff-format
      - id: ruff-check
        args: [--fix]

  # --- Security ---
  - repo: https://github.com/pre-commit/pre-commit-hooks
    rev: v6.0.0
    hooks:
      - id: detect-private-key

  - repo: https://github.com/PyCQA/bandit
    rev: 1.9.4
    hooks:
      - id: bandit
        args: [-r, src/]
        pass_filenames: false

  # --- Type checking (local — no official pre-commit repo for ty yet) ---
  - repo: local
    hooks:
      - id: ty
        name: ty type check
        entry: uv run --with ty ty check
        language: system
        types: [python]
        pass_filenames: false

  # --- Docstring coverage ---
  - repo: https://github.com/econchick/interrogate
    rev: 1.7.0
    hooks:
      - id: interrogate
        pass_filenames: false

  # --- Documentation regeneration ---
  - repo: local
    hooks:
      - id: pdoc
        name: regenerate API docs
        entry: uv run --with pdoc pdoc --docformat google src/requirements_agent_tools -o docs/
        language: system
        pass_filenames: false
        always_run: true

  # --- Tests ---
  - repo: local
    hooks:
      - id: pytest
        name: pytest
        entry: uv run pytest
        language: system
        pass_filenames: false
        always_run: true
```

**Note on bandit args:** `bandit` from PyCQA's pre-commit hook passes filenames by default (`pass_filenames: true` in their `.pre-commit-hooks.yaml`). Adding `args: [-r, src/]` and `pass_filenames: false` overrides this to scan the full `src/` tree consistently regardless of which files are staged. [VERIFIED: PyCQA/bandit `.pre-commit-hooks.yaml`]

**Note on ruff hook IDs:** The hook IDs are `ruff-format` and `ruff-check` (not `ruff` — that was the old single-hook name). [VERIFIED: astral-sh/ruff-pre-commit README]

### Pattern 3: `[tool.interrogate]` in pyproject.toml

**What:** interrogate reads its config from `[tool.interrogate]` in `pyproject.toml`. When the pre-commit hook runs without explicit `--fail-under` args, it uses the config file value. [VERIFIED: interrogate.readthedocs.io]

**Current docstring coverage: 51.2%** — well below the 90% target. This means the planner must include a wave of docstring-writing tasks before the interrogate hook can be enabled without blocking every commit.

```toml
# Source: interrogate.readthedocs.io — verified exact field names and types
[tool.interrogate]
fail-under = 90
ignore-init-method = true
ignore-init-module = true
ignore-magic = true
ignore-private = true
ignore-semiprivate = false
verbose = 0
quiet = false
color = true
exclude = ["tests", "docs"]
```

**Why ignore-init-method and ignore-init-module = true:** `__init__` methods and `__init__.py` module docstrings typically have no meaningful content to document; excluding them avoids noise and lets the 90% threshold focus on actual public API functions. [ASSUMED — common interrogate convention; adjust to user preference]

**pre-commit hook — no explicit `--fail-under` needed** when `pyproject.toml` has `[tool.interrogate]` config, because interrogate reads it automatically. The pre-commit hook just runs `interrogate src/`. `pass_filenames: false` is required for config-file-based settings to apply correctly. [VERIFIED: interrogate.readthedocs.io]

### Anti-Patterns to Avoid

- **`uvx pdoc ...` as pre-commit hook:** Fails because uvx runs in an isolated environment without project deps (loguru, pydantic, sqlite-vec not available). Use `uv run --with pdoc pdoc ...` instead. [VERIFIED: confirmed failure via direct testing]
- **`pdoc --format markdown`:** This flag does not exist. pdoc is HTML-only. [VERIFIED: `pdoc --help`]
- **`pass_filenames: true` on ty/interrogate/pdoc/pytest:** These tools need whole-project context, not a list of changed files. Always use `pass_filenames: false`. [CITED: interrogate docs + pre-commit local hook patterns]
- **Adding dev tools to `[project.dependencies]`:** `pdoc`, `pre-commit`, `ty`, `bandit`, `interrogate` are dev-only. Use `uv add --dev <tool>` — this writes to `[dependency-groups] dev` in pyproject.toml (uv's convention for dev deps). [ASSUMED: uv dev group convention — verify exact pyproject.toml key with `uv add --dev --dry-run`]
- **Relying on `ruff` hook ID:** The old single `ruff` hook id was split into `ruff-format` and `ruff-check` in recent versions of ruff-pre-commit. Use the two-hook form. [VERIFIED: astral-sh/ruff-pre-commit README]

## Don't Hand-Roll

| Problem | Don't Build | Use Instead |
|---------|-------------|-------------|
| API docs from docstrings | Custom doc generator | `pdoc` — handles subpackages, cross-links, search |
| Docstring coverage % | File-walking counter | `interrogate` — handles nested classes, magic methods, inheritance |

## Critical Open Question — D-10 Markdown Output

**Decision D-10** states: "Generate both HTML and Markdown output formats."

**Research finding:** pdoc 16.0.0 does not support Markdown output. The output format is HTML-only. A GitHub issue requesting Markdown output (mitmproxy/pdoc#308) was closed as "not planned." There is no `--format` flag, no Markdown template available out-of-the-box, and renaming HTML files to `.md` produces broken output (confirmed by issue discussion). [VERIFIED: pdoc --help + issue #308]

**Options for the planner to discuss with user before locking this task:**

1. **HTML-only (recommended):** Drop the Markdown requirement. `docs/` contains only HTML. This is what pdoc supports natively — zero extra work.
2. **Add `lazydocs` for Markdown alongside pdoc:** `lazydocs` generates GitHub-flavored Markdown from Google-style docstrings. Would produce a parallel `docs/md/` tree. Adds a second tool and second pre-commit hook.
3. **Custom pdoc Jinja2 template:** Write a `module.html.jinja2` that outputs Markdown-like content. Fragile, unsupported, maintenance burden.

**Recommendation:** Option 1 (HTML-only). The committed `docs/` HTML is browsable locally and via GitHub Pages if desired. Markdown doc files offer no practical advantage for this agent-facing codebase.

## Common Pitfalls

### Pitfall 1: pdoc runs in wrong environment
**What goes wrong:** `uvx pdoc src/...` exits with `ModuleNotFoundError: No module named 'loguru'` (or any other project dep).
**Why it happens:** `uvx` creates an isolated environment containing only pdoc itself. The project's dependencies are not present.
**How to avoid:** Always invoke pdoc as `uv run --with pdoc pdoc ...` — this adds pdoc to the project's existing virtual environment where all deps are available.
**Warning signs:** Any `ModuleNotFoundError` or `ImportError` when pdoc runs.

### Pitfall 2: interrogate coverage at 51.2% blocks commits immediately
**What goes wrong:** Enabling the interrogate hook with `fail-under = 90` will immediately fail every commit because current coverage is 51.2%.
**Why it happens:** Most functions in the codebase lack Google-style docstrings.
**How to avoid:** The planner should sequence docstring-writing tasks in a wave that runs *before* the wave that enables the interrogate hook. The hook can be added to `.pre-commit-config.yaml` with an initial `fail-under = 0` (or commented out) and raised to 90 after the docstring wave completes. Alternatively, use `git commit --no-verify` during the docstring-writing wave (acceptable for a scaffold phase).
**Warning signs:** Every commit failing with `RESULT: FAILED (minimum: 90.0%, actual: 51.2%)`.

### Pitfall 3: ty produces errors on existing code (sqlite3.Row access)
**What goes wrong:** `ty check src/` reports `invalid-argument-type` errors in `db/_serialization.py` on sqlite3.Row dict-style access.
**Why it happens:** sqlite-vec does not ship type stubs; ty infers `Any` from sqlite3.Row, which it conservatively treats as potentially a tuple.
**How to avoid:** Set `invalid-argument-type = "warn"` in `[tool.ty.rules]` initially, or add `# ty: ignore[invalid-argument-type]` inline comments to the affected lines in `_serialization.py`.
**Warning signs:** Commit blocked by type errors in `db/_serialization.py`.

### Pitfall 4: bandit flags security patterns in existing code
**What goes wrong:** bandit finds 3 medium-severity issues in `src/` on the current codebase.
**Why it happens:** bandit's default profile is conservative — it flags patterns like `subprocess`, `assert`, and certain exception-handling styles.
**How to avoid:** Review the 3 flagged issues before enabling the bandit hook. Either fix the issues or add `# nosec` inline comments with justification. Use `uv run --with bandit bandit -r src/` to review them before the hook is active.
**Warning signs:** Commit blocked by `Issue: [B...] ...` bandit output.

### Pitfall 5: pdoc pre-commit hook writes docs/ but changes aren't committed
**What goes wrong:** The pdoc hook regenerates `docs/` on every commit, but the newly written files are not staged — so the committed snapshot may lag behind the code.
**Why it happens:** pre-commit hooks run after `git add` has already captured the staging area. Files written by a hook during the commit are not automatically staged.
**How to avoid:** Either (a) run `uv run --with pdoc pdoc ... && git add docs/` manually before committing, or (b) document that developers must run `git add docs/` after the hook modifies docs. For the scaffold phase, adding a note in CLAUDE.md is sufficient.
**Warning signs:** `git status` shows modified `docs/` files after a commit.
